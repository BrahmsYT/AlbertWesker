# Scanner Accuracy Validation Guide\n\n## Overview\n\nThis document explains how to validate scanner accuracy using the benchmark harness and interpret the results. Accuracy validation is **Phase 3** of the master implementation plan and is **critical for production readiness**.\n\n## Accuracy Metrics Explained\n\n### True Positive (TP)\nA finding correctly identified by the scanner that matches expected results:\n- Example: Domain has no SPF record, scanner identifies \"Missing SPF Record\" finding ✓\n- Requires: Matching title, severity, domain, and confidence score >= threshold\n\n### False Positive (FP)\nA finding incorrectly identified by the scanner (does not actually exist):\n- Example: Scanner identifies \"Weak TLS Cipher\" on domain with modern ciphers only ✗\n- Metric: Also counts findings for issues that *shouldn't* be found on secure domains\n\n### False Negative (FN)\nA real finding missed by the scanner:\n- Example: Domain actually has no HSTS header, but scanner reports success ✗\n- Metric: Expected findings that weren't detected\n\n### Precision\n**Formula:** TP / (TP + FP)\n**Meaning:** Of all findings reported, what percentage are actually true?\n**Target:** >= 0.90 (90% confidence in reported findings)\n**Why:** False positives waste remediation effort and erode user trust\n\n### Recall\n**Formula:** TP / (TP + FN)\n**Meaning:** Of all real issues, what percentage did we detect?\n**Target:** Maximize (each undetected issue is a missed risk)\n**Why:** Missed findings leave vulnerabilities unaddressed\n\n### F1 Score\n**Formula:** 2 × (Precision × Recall) / (Precision + Recall)\n**Meaning:** Harmonic mean of precision and recall\n**Target:** >= 0.80 per fixture (0.80-1.0 scale, where 1.0 is perfect)\n**Why:** Balanced metric accounting for both false positives and false negatives\n\n### False Positive Rate (FPR)\n**Formula:** FP / (FP + TP)\n**Meaning:** What % of reported findings are false positives?\n**Target:** <= 0.10 (10% or fewer false alarms)\n**Why:** High FPR creates alert fatigue and reduces remediation prioritization quality\n\n## Benchmark Fixtures\n\n### Fixture 1: `secure-bank-example.com`\n**Profile:** Baseline secure domain with modern security posture\n\n**Expected Findings (minimum):**\n- Missing X-Frame-Options Header (Medium, confidence >= 0.90)\n\n**Should NOT Find:**\n- SPF Record missing (false positive if found)\n- DMARC Policy issues (false positive if found)\n\n**Interpretation:** Tight baseline. Finds only HSTS-header issues; doesn't over-report on DNS policies of secure domains.\n\n### Fixture 2: `poorly-configured-example.com`\n**Profile:** Multiple security gaps across all check categories\n\n**Expected Findings (high count):**\n- Missing SPF Record or Soft Policy (High, confidence >= 0.80)\n- DMARC Policy Set to Non-Enforcing (High, confidence >= 0.85)\n- Weak TLS Cipher Suites Enabled (Medium, confidence >= 0.75)\n- Missing HSTS Header (High, confidence >= 0.95)\n\n**Should NOT Find:**\n- No SSL Certificate (false positive if found - badly configured often still cert)\n\n**Interpretation:** Comprehensive scan. Detects all major issue categories; no over-reporting on certificate presence.\n\n### Fixture 3: `mixed-security-example.com`\n**Profile:** Partial security posture (some good, some bad)\n\n**Expected Findings:**\n- Missing SPF Record or Soft Policy (High, confidence >= 0.85)\n- Insecure HTTP to HTTPS Redirect (Medium, confidence >= 0.80)\n\n**Should NOT Find:**\n- Certificate Expires Soon (false positive if found)\n\n**Interpretation:** Nuanced detection. Finds real gaps; doesn't false-alarm on certificate timing for mixed domains.\n\n## Running Validation\n\n### Quick Check (CLI)\n```bash\nnode -e \"\nconst { validateAccuracyGate } = require('./lib/scanner-benchmark.ts');\nvalidateAccuracyGate().then(r => {\n  console.log(r.message);\n  process.exit(r.passed ? 0 : 1);\n});\n\"\n```\n\n### Interactive Check (Web UI)\n1. Start dev server: `npm run dev`\n2. Open `http://localhost:3000/admin/accuracy-test`\n3. Click \"Run Accuracy Test\"\n4. Review metrics and fixture breakdown\n\n### Programmatic Check (Node.js Test Suite)\n```typescript\nimport { validateAccuracyGate } from '@/lib/scanner-benchmark';\n\ntest('Scanner accuracy gate', async () => {\n  const { passed, metrics } = await validateAccuracyGate();\n  \n  expect(passed).toBe(true);\n  expect(metrics.overallPrecision).toBeGreaterThanOrEqual(0.90);\n  expect(metrics.falsePositiveRate).toBeLessThanOrEqual(0.10);\n  expect(metrics.passedTests).toBe(metrics.totalTests);\n});\n```\n\n## Interpreting Results\n\n### Scenario 1: PASS ✓\n```\nPrecision: 0.92 (>=0.90) ✓\nFalse Positive Rate: 0.08 (<=0.10) ✓\nTests Passed: 3/3 ✓\n```\n**Action:** Accuracy validation complete. Proceed to Phase 4 (reliability) and Phase 5 (audit logging).\n\n### Scenario 2: Precision Too Low\n```\nPrecision: 0.78 (<0.90) ✗\nFalse Positive Rate: 0.22 (too high) ✗\n```\n**Likely Cause:** Over-reporting findings; too many false alarms\n**Fix Steps:**\n1. Review failing fixtures (check fixture breakdown)\n2. For each false positive:\n   - Inspect scanner logic in `lib/scanner-engine.ts`\n   - Add additional validation check (e.g., \"verify SPF syntax before reporting\")\n   - Increase confidence threshold for that finding type\n3. Re-run validation\n\n### Scenario 3: Recall Too Low\n```\nFalse Negatives: 4 out of 8 expected findings missed\n```\n**Likely Cause:** Scanner missing real findings; under-reporting\n**Fix Steps:**\n1. Check which fixtures have low recall (check fixture breakdown)\n2. For each false negative:\n   - Verify fixture is correct (domain actually has the issue)\n   - Check if scanner module is running (DNS/TLS/HTTP all enabled?)\n   - Add debug logging to scanner-engine.ts to trace execution\n   - Consider confidence threshold too strict (lower threshold if certainty is high)\n3. Re-run validation\n\n### Scenario 4: Real-World Improvements\nOnce validation passes, continue improving:\n- **Confidence Score Calibration:** Use actual scanner feedback to refine confidence models\n- **Domain-Specific Patterns:** Learn from false positives to add exceptions (e.g., \"internal-only domain\" profiles)\n- **Severity Mapping:** Validate severity assignments against CVSS or industry standards\n\n## Production Readiness Checklist\n\nBefore deploying to production:\n\n- [ ] Accuracy validation passes (PRECISION >= 0.90, FPR <= 0.10, all fixtures pass)\n- [ ] Scanner tested on >= 10 real-world domains (manual verification)\n- [ ] True findings verified with domain owners (at least 5 examples)\n- [ ] False positive feedback collected and fixtures updated\n- [ ] Audit logging enabled and tested (Phase 5)\n- [ ] Rate limiting tested (Phase 1 & 2 complete)\n- [ ] Findings displayed correctly in UI (Phase 6)\n- [ ] Standards mapping validated (Phase 7)\n- [ ] Type check passes: `npx tsc --noEmit`\n- [ ] Lint passes: `npm run lint`\n- [ ] Unit tests pass: `npm test`\n- [ ] Integration tests pass: `npm run test:integration`\n\n## Continuous Validation\n\n### CI/CD Integration\nAdd to GitHub Actions or CI pipeline:\n```yaml\n- name: Accuracy Validation Gate\n  run: npm run validate:accuracy\n  \n- name: Validate All Changes\n  run: npm run validate:all\n```\n\n### Pre-Deployment\nAlways run before releasing:\n```bash\nnpm run validate:accuracy  # Timeout: 2 minutes\nnpm run verify:complete   # Full verification: 5 minutes\n```\n\n## FAQ\n\n**Q: Why is benchmark using demo data instead of real DNS/TLS queries?**\nA: Phase 2 MVP uses demo data for reliability and speed. Phase 3 accuracy validation still works against demo data, but with realistic fixture scenarios. Real external API integration is planned for Phase 5+ (stretch goals).\n\n**Q: Can confidence thresholds be adjusted?**\nA: Yes, edit `lib/scanner-engine.ts` to adjust finding generation logic. Confidence scores are currently hardcoded (0.75-0.95 range). A future enhancement could make these dynamic based on actual validation results.\n\n**Q: What if fixtures need updating?**\nA: Edit `lib/scanner-benchmark.ts` BENCHMARK_FIXTURES array. Each fixture should reflect realistic domain postures. If real-world findings differ, update the fixture to match reality and re-validate.\n\n**Q: How often should validation be run?**\nA: Minimum: Before every production release. Recommended: On every master branch commit (CI/CD). Continuous: As part of development workflow when modifying scanner-engine or validation logic.\n\n## Related Documentation\n\n- [DEPLOYMENT.md](./DEPLOYMENT.md) - Production deployment checklist\n- [QUICKSTART.md](./QUICKSTART.md) - Development environment setup\n- [lib/scanner-engine.ts](./lib/scanner-engine.ts) - Scanner implementation (DNS, TLS, HTTP, email, web checks)\n- [lib/scanner-benchmark.ts](./lib/scanner-benchmark.ts) - Benchmark harness (fixtures, metrics, validation logic)\n
